# Analytics Engine

## Philosophy

The user opens the app and sees a clear picture of their money — how much came in, how much went out, where it went, and whether they're doing better or worse than last week. No charts that require interpretation. No jargon. Plain English.

Everything on the home screen should answer one of:
- "Am I spending more or less than usual?"
- "Where is my money going?"
- "What's coming up?"

## 1. What Gets Computed vs What Gets Queried Live

The critical decision: **pre-compute weekly summaries, compute everything else live**.

| Data | Strategy | Why |
|---|---|---|
| Weekly summaries | **Pre-computed** (batch, Monday 03:00) | Historical, never changes once the week ends |
| Current week totals | **Live** (query on request) | Still accumulating, must be fresh |
| Month-to-date totals | **Live** | Same — still accumulating |
| Category breakdown | **Live** | Depends on the date range the user picks |
| Budget spend tracking | **Live** | Must reflect the latest transaction |
| Recurring payments | **Pre-computed** (after each sync) | Pattern detection is expensive |
| Insight cards | **Pre-computed** (batch + after sync) | Generated by rules, delivered as-is |
| Top merchants | **Live** | Simple aggregation, fast enough |
| Fixed vs flexible | **Live** (using recurring_payments table) | Combines pre-computed recurring data with live totals |

### Why not pre-compute everything?

Pre-computing month-to-date or current-week means the number is stale the moment a new transaction syncs. For a personal finance app, "I just got paid and the app doesn't show it" is unacceptable. The queries are fast because they're scoped to one user and indexed on `(userId, bookedAt)`.

### Why pre-compute weekly summaries at all?

Because comparing "this week vs last 12 weeks" means computing 12 weeks of data on every request. Pre-computing past weeks means the comparison is a simple DB read. Current week is always live.

## 2. Data Model

### Pre-computed: `weekly_summaries` (already exists)

```
| Field                  | Type          | Purpose                                    |
|------------------------|---------------|--------------------------------------------|
| userId + weekStart     | UNIQUE        | One row per user per week                  |
| totalSpent             | DECIMAL(12,2) | Sum of abs(amount) where amount < 0        |
| totalEarned            | DECIMAL(12,2) | Sum of amount where amount > 0             |
| netFlow                | DECIMAL(12,2) | totalEarned - totalSpent                   |
| transactionCount       | INT           | Number of transactions in the week         |
| topCategoryId          | TEXT          | Category with highest spend                |
| topCategoryAmount      | DECIMAL(12,2) | How much was spent in that category        |
| categoryBreakdown      | JSONB         | { "GROCERIES": 142.50, "DINING": 87.30 }  |
| comparedToPrevWeekPct  | DECIMAL(5,2)  | % change vs previous week's totalSpent     |
```

### Pre-computed: `insights` (already exists)

```
| Field     | Type     | Purpose                                           |
|-----------|----------|---------------------------------------------------|
| type      | TEXT     | Rule that generated this insight                  |
| title     | TEXT     | Headline for the card (plain English)             |
| body      | TEXT     | One-sentence explanation                          |
| data      | JSONB    | Structured data for the mobile UI to render       |
| priority  | INT 0-10 | Higher = more prominent on home screen            |
| expiresAt | DATETIME | Auto-cleanup for time-sensitive insights          |
```

### Live-computed: returned directly from API, not stored

- Current week totals (spending, earnings, net flow)
- Month-to-date totals
- Category breakdown for any date range
- Budget current spend
- Top merchants for a period
- Fixed vs flexible breakdown

## 3. Batch Job Design

### Job: Weekly Summary Generation

**Schedule:** Monday 03:00 UTC (covers Mon–Sun of the previous week)
**Queue:** `notify:weekly`

```
For each active user (not soft-deleted):
  1. Compute Monday 00:00 → Sunday 23:59 of previous week
  2. Query transactions in that range
  3. Calculate: totalSpent, totalEarned, netFlow, transactionCount
  4. Calculate: categoryBreakdown (group by categoryId, sum abs(amount))
  5. Find topCategory (highest spend)
  6. Load previous week's summary → compute comparedToPrevWeekPct
  7. Upsert into weekly_summaries
  8. Run insight generation for this week's data
  9. Create a WEEKLY_DIGEST notification
```

### Job: Budget Snapshot + Alerts

**Schedule:** Daily 02:00 UTC (or after each sync)
**Queue:** `notify:budget`

```
For each active budget:
  1. Compute current period start/end based on budget.period and periodStartDay
  2. Sum spending in that category for the period
  3. If period just ended → create BudgetSnapshot row
  4. If spend >= alertAtPercent of limit → create BUDGET_WARNING insight
  5. If spend > limit → create BUDGET_EXCEEDED insight
```

### Job: Insight Cleanup

**Schedule:** Daily 04:00 UTC

```
DELETE FROM insights WHERE expires_at < NOW()
```

## 4. Weekly Summary Generation (Detail)

```typescript
async function computeWeeklySummary(userId: string, weekStart: Date): WeeklySummaryData {
  const weekEnd = addDays(weekStart, 6); // Sunday 23:59:59

  const transactions = await prisma.transaction.findMany({
    where: { userId, bookedAt: { gte: weekStart, lte: endOfDay(weekEnd) } },
    select: { amount: true, categoryId: true, merchantId: true },
  });

  let totalSpent = 0, totalEarned = 0;
  const categoryTotals: Record<string, number> = {};

  for (const tx of transactions) {
    const amt = Number(tx.amount);
    if (amt < 0) {
      totalSpent += Math.abs(amt);
      categoryTotals[tx.categoryId] = (categoryTotals[tx.categoryId] ?? 0) + Math.abs(amt);
    } else {
      totalEarned += amt;
    }
  }

  const [topCategoryId, topCategoryAmount] = Object.entries(categoryTotals)
    .sort(([, a], [, b]) => b - a)[0] ?? [null, null];

  // Compare to previous week
  const prevSummary = await prisma.weeklySummary.findUnique({
    where: { userId_weekStart: { userId, weekStart: subDays(weekStart, 7) } },
  });
  const prevSpent = prevSummary ? Number(prevSummary.totalSpent) : null;
  const comparedToPrevWeekPct = prevSpent && prevSpent > 0
    ? round(((totalSpent - prevSpent) / prevSpent) * 100, 1)
    : null;

  return {
    totalSpent, totalEarned,
    netFlow: totalEarned - totalSpent,
    transactionCount: transactions.length,
    topCategoryId, topCategoryAmount,
    categoryBreakdown: categoryTotals,
    comparedToPrevWeekPct,
  };
}
```

## 5. Insight Generation Rules

Insights are plain-English cards on the home screen. They come in two flavours:

**Post-weekly-summary insights** — generated Monday after the batch job:

| Type | Rule | Example |
|---|---|---|
| `SPENDING_UP` | comparedToPrevWeekPct > 20 | "You spent 34% more than last week — mostly on Dining." |
| `SPENDING_DOWN` | comparedToPrevWeekPct < -15 | "Nice work! You spent 22% less than last week." |
| `BIG_CATEGORY` | One category > 40% of total spend | "Groceries made up 45% of your spending this week." |
| `LOW_INCOME_WEEK` | totalEarned < 100 and totalEarned < prevWeek | "Quiet week for income — only €42 came in." |
| `HIGH_SPEND_DAY` | Any single day > 30% of week total | "Wednesday was your biggest day — €187 spent." |

**Post-sync insights** — generated after transaction data arrives:

| Type | Rule | Example |
|---|---|---|
| `BUDGET_WARNING` | Category spend >= alertAtPercent of limit | "Heads up — you've used 82% of your Dining budget." |
| `BUDGET_EXCEEDED` | Category spend > limit | "You've gone over your €200 Groceries budget by €23." |
| `UNUSUAL_MERCHANT` | Amount > 3x average for that merchant | "Your Deliveroo order was €67 — usually it's around €22." |
| `NEW_SUBSCRIPTION` | RecurringDetector finds new recurring | "Looks like you started a new subscription to ChatGPT (€20/mo)." |
| `SUBSCRIPTION_INCREASE` | Recurring amount increased > 10% | "Your Netflix went up from €12.99 to €15.99." |
| `SUBSCRIPTION_GONE` | Recurring payment missed and not dismissed | "We haven't seen your Disney+ charge — it may have been cancelled." |
| `CONSENT_EXPIRING` | Bank connection expires within 7 days | "Your AIB connection expires in 5 days. Tap to renew." |

**Priority rules:**
- Budget exceeded: 9
- Consent expiring: 8
- Spending up significantly: 7
- New subscription: 6
- Budget warning: 5
- Everything else: 3

**Expiry rules:**
- Weekly digest insights: expire after 7 days
- Budget alerts: expire at end of budget period
- Consent alerts: expire after connection renewed or expired

## 6. Fixed vs Flexible Spending

This is a derived metric, not a stored one. The split:

| Type | Source | Example |
|---|---|---|
| **Fixed** | `recurring_payments` where `isActive = true` | Rent, Netflix, electricity, gym membership |
| **Flexible** | Total spend minus fixed spend | Groceries, dining, shopping, entertainment |

```typescript
// Compute for current month
const recurringMonthly = await getMonthlyRecurringTotal(userId); // from recurring_payments
const totalSpentThisMonth = await getMonthToDateSpending(userId);
const fixedSpend = recurringMonthly;
const flexibleSpend = totalSpentThisMonth - fixedSpend;
```

This gives users a simple mental model: "€840 of my spending this month is locked in (rent, bills, subscriptions). I have control over the other €420."
